LIDHRA / concept

Product & design concept · a modern cross-platform download manager

Link what matters.

Lidhra is a modern, cross-platform remote manager for your downloads and data. One battle-tested Rust core, refracted into six shells that each speak their platform's native design language — Liquid Glass on Apple, Material 3 on Android, Fluent on Windows, libadwaita on Linux — wrapped in a Tauri shell that ships in single-digit megabytes. From the Albanian lidh — to connect, to bind.

◆ Tauri + Rust core ◆ Liquid Glass · Material 3 · Fluent · libadwaita ◆ iOS = App-Store-safe remote ◆ ~6 MB installer

04 · Architecture

Why Tauri + Rust makes this possible — and tiny

One Rust core, many thin native fronts. No bundled browser engine means installers in single-digit megabytes and low idle RAM — the resource win P2 asked for.

Shell · UI
Per-platform webview rendered by the OS (WKWebView / WebView2 / WebKitGTK). The design-language layer lives here: Liquid Glass, Material 3, Fluent, libadwaita — CSS themes over one component set.
system webviewno Chromium~600 KB UI
Lidhra core
Rust orchestration: window/tray/menu-bar management, IPC command bus, session state, theming tokens, secure remote transport (WireGuard-friendly). Compiled per target, shared across all platforms.
Tauri 2Rusttauri-plugin-*mobile targets
Engine
The proven transfer engine — libtorrent-rasterbar via FFI (or a Rust reimplementation long-term). Reused, not rewritten: the part users already trust stays intact.
libtorrentFFIopen-source engine
Remote
The core exposes / consumes a standard Web API, so the iOS PWA, the watch glance, and any "remote mode" client talk to a headless box on a Pi/NAS with zero new server software.

05 · Store compliance & the piracy question

Passing review by designing for the rules, not around them

The hard constraint isn't technical — it's policy. Apple bans download/torrent clients and remote controllers on anti-piracy grounds. Lidhra's distribution strategy is different per store, by design — and the "remote manager for downloads & data" positioning is content-neutral from the name down.

iOS App Store — ✕ don't submit

Apple rejects native torrent clients and remote apps outright. Don't fight this. Ship the iOS + watch experience as an installable PWA served from the user's own box — Liquid-Glass UI, Home-Screen icon, no review queue, no rejection.

macOS — ⚠ notarize, skip MAS

Distribute directly, Developer-ID signed & notarized (Gatekeeper-clean), not via the sandbox-heavy Mac App Store. This is how Transmission, Deluge, and qBittorrent already ship on Mac.

Play Store — ✓ allowed

Google permits download clients and remotes. Ship the full Material 3 app. Keep it content-neutral: no built-in indexes of infringing content, no piracy-adjacent marketing.

The content-neutrality doctrine (applies everywhere)

✓ What keeps it legitimate
  • Ships empty — a general-purpose file-transfer tool, like the OS's own downloader.
  • Default and demo content is Linux ISOs / Creative-Commons — never copyrighted media.
  • No bundled search indexes of infringing sites; search is opt-in, user-provided.
  • Signed, notarized, reproducible builds → answers "is this the real Lidhra?" (P3 trust finding).
✕ What gets you rejected / delisted
  • Marketing that references piracy, "free movies," or specific copyrighted titles.
  • Preloaded trackers or indexes pointing at infringing content.
  • Any App Store submission of a torrent/remote binary for iOS.
  • Bundling a VPN "to hide your downloading" — reframes the tool as piracy-oriented.
